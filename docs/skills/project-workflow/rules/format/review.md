# 审查评分规格

基于 start-review 统一评分框架，针对 spec/design/plan/release 的评审标准。

## 评分标准（1-5）

| 分数 | 含义 |
|------|------|
| 5 | 卓越，可作范例 |
| 4 | 良好，无明显问题 |
| 3 | 合格，有改进空间 |
| 2 | 不足，需修改 |
| 1 | 严重不足，需重做 |

## 发现分级

| 级别 | 含义 | 处理 |
|------|------|------|
| **P0** | 阻塞——缺少关键章节、逻辑矛盾、上游映射遗漏 | 必须修复 |
| **P1** | 严重——约束不可验证、verify 不可执行、边界不清晰 | 必须修复 |
| **P2** | 改进——信息密度低、措辞模糊但可推断 | 可带 TODO 通过 |
| **P3** | 建议——格式优化、补充示例 | 不阻塞 |

## 决策规则

| 条件 | 决策 |
|------|------|
| 无 P0/P1 且均分 ≥ 4.0 | ✅ **PASS** |
| 无 P0，P1 ≤ 2 条，均分 ≥ 3.5 | ⚠️ **CONDITIONAL**（标注修复项） |
| 存在 P0，或 P1 > 2 条，或均分 < 3.5 | ❌ **FAIL** |

## 各文档评审维度

### spec.md

| 维度 | 权重 | 评估要点 |
|------|------|---------|
| 边界清晰 | 高 | Out of Scope 是否明确？ |
| 可测试性 | 高 | 验收标准能否写成自动化测试？ |
| 完备性 | 中 | 异常场景覆盖正常流每步失败？ |
| 职责正确 | 中 | 是否混入技术实现？ |

### design.md

| 维度 | 权重 | 评估要点 |
|------|------|---------|
| 映射完整 | 高 | spec 每个要素都有对应？ |
| 约束可验证 | 高 | 每条约束有量化指标？ |
| 影响范围准确 | 中 | 能否直接拆任务？ |
| 职责正确 | 中 | 是否混入 plan 或 spec 内容？ |

### plan.md

| 维度 | 权重 | 评估要点 |
|------|------|---------|
| 映射完整 | 高 | design 影响范围每行有对应 Task？ |
| 可执行 | 高 | verify 可执行？scope 精确到文件？ |
| 依赖正确 | 中 | depends_on 正确无循环？ |
| 风险覆盖 | 中 | 关键风险有缓解措施？ |

### release.md

| 维度 | 权重 | 评估要点 |
|------|------|---------|
| 完整性 | 高 | 需求表与 active/ 已完成需求一致？ |
| 可追溯 | 高 | 每个需求有关联 commits？ |
| 验证状态 | 中 | 所有 checkbox 已勾选？ |
| frontmatter | 中 | version/date/retain_until 合法？ |

## 输出格式

```markdown
## 审查报告: {文件名}

| 维度 | 分数 | 说明 |
|------|------|------|
| {维度} | {N}/5 | {一句话} |

**均分:** {X}/5.0
**决策:** ✅ PASS / ⚠️ CONDITIONAL / ❌ FAIL

### 发现
- **P{N}** {位置} — {问题} → {修复建议}
```
