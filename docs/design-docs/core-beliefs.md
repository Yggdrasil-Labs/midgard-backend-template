# Core Beliefs

状态：active

适用范围：整个 Midgard Backend Template 的文档组织、协作方式与长期设计信念

最后验证：2026-04-11，已与当前 `AGENTS.md`、`ARCHITECTURE.md` 和 `docs/` 结构对齐
	
本文件把 OpenAI 那篇“harness engineering”的核心思想翻译成适用于 Midgard 后端模板的长期信念。

## 1. 仓库是记录系统

实现、计划、质量约束和已知债务都应该回到仓库中，以版本化文件存在。

原因：

- Agent 的上下文是稀缺资源。
- 口头说明、聊天记录和一次性 prompt 很难长期复用。
- 版本化文档比“记忆”更适合团队协作与后续维护。

## 2. `AGENTS.md` 负责导航，不负责包办一切

单个超长手册会很快失效。

因此：

- `AGENTS.md` 只保留稳定地图、硬约束和阅读顺序。
- 真实细节下沉到专题文档。
- 让 Agent 通过索引逐步展开，而不是一开始就吞下全部背景。

## 3. 架构清晰优先于实现速度

在这个模板中，最大的放大器不是“多写一点代码”，而是让 Agent 始终知道：

- 改动属于哪一层
- 依赖是否越界
- 对象应该如何流动
- 哪些名字和目录是稳定约定

一旦这些结构不清晰，后续生成的代码质量会快速退化。

## 4. 文档要服务执行，而不是装饰仓库

文档存在的目标是让下一个 Agent 继续工作，而不是让首页看起来完整。

因此每份文档都应回答至少一个问题：

- 我现在应该改哪里
- 这个规则为什么存在
- 这个计划进行到哪了
- 现在最重要的风险是什么

如果答不上这些问题，文档很可能只是噪音。

## 5. 计划是第一等工件

中大改动应当有执行计划，并记录：

- 目标
- 范围
- 决策
- 验证方式
- 未完成项

计划不是为了流程感，而是为了让任务可以被中断、接手和继续。

## 6. 生成内容和人工判断要分层

本仓库允许 `docs/generated/` 这类目录存在，用于承载：

- 数据库结构导出
- 代码扫描结果
- 自动化生成的摘要

但这些内容不应替代人工判断。人工总结应进入专题文档或计划文档，并显式说明判断依据。

## 7. 质量需要被显式评分

智能体吞吐提高之后，瓶颈往往变成：

- 文档是否可信
- 架构是否一致
- 风险是否被看见
- 测试是否覆盖关键路径

所以质量不能只靠“感觉不错”，而要通过可维护的评分卡和检查项来暴露差距。

## 8. 模板仓库也需要前瞻性约束

Midgard 当前是模板，不是成熟业务系统。

这意味着文档应明确区分：

- 已落地事实
- 推荐模板
- 未来扩展位
- 当前缺口

避免把“还没实现的方向”写成“已经成立的能力”。
